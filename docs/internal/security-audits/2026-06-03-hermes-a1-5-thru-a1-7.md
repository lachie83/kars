# Security Audit — Hermes runtime A1.5 + A1.6 + A1.7 + http_fetch

**Scope**: Implements the kars tool catalogue inside the Hermes plugin — sub-agent spawn family, peer discovery, Foundry memory wrapper, HTTP fetch. Builds on `e105350` (A1.3+A1.4).

## Changes

### 1. `runtimes/hermes/src/kars_runtime_hermes/plugin/spawn.py` (A1.5)

Replaces the A1.1 stub with the full `kars_spawn` family — 4 tools:
- **`kars_spawn`** — POST `/sandbox/spawn`, polls `/sandbox/{name}/status` for up to 45s waiting for `Running`. Validates `name` against K8s metadata.name rules (DNS-label, ≤63 chars). Honors `KARS_DEV_PROFILE=true` by injecting `learn_egress: true`. Surfaces the Hermes Act 1 mesh limitation in the result message so the LLM knows to use Foundry Memory Store / Conversations as the coordination channel.
- **`kars_spawn_status`** — GET `/sandbox/{name}/status`. 404 → "not found" error.
- **`kars_spawn_destroy`** — DELETE `/sandbox/{name}` + DELETE `/agt/trust/{name}` (best-effort trust cleanup). 404 → "already gone" warning.
- **`kars_spawn_list`** — GET `/sandbox/list`.

12 new tests covering name validation, router body shape, dev-profile flag injection, status / destroy / list paths, registration.

### 2. `runtimes/hermes/src/kars_runtime_hermes/plugin/discover.py` (A1.6)

Replaces the A1.1 stub with `kars_discover`:
- Direct lookup when called with `did:` (`GET /agt/registry/v1/agents/{did}`).
- Search by display name when called with `name:` (`GET /agt/registry/v1/agents?display_name=X`).
- Returns the full registry record (AMID, capabilities, reputation, tier, verified Entra app ID if applicable).

7 new tests covering both paths, 404 handling, empty-result case, schema registration.

### 3. `runtimes/hermes/src/kars_runtime_hermes/plugin/foundry.py` (A1.7)

**Design decision**: The router already exposes all 9 Foundry tools at `POST /platform/mcp`. Hermes ships strong native MCP. So we leverage that for 8 of the 9 tools instead of re-implementing them, and add ONE native wrapper for `foundry_memory` because:
1. It's the most-used Foundry tool by far
2. It honors the kars `memory-${SANDBOX_NAME}` convention (per contract §KarsMemory binding)
3. Sub-second latency vs round-tripping through the MCP gateway

`foundry_memory` supports 3 operations:
- **`search`** — POST `/memory_stores/{store}:search_memories` with `query_text + scope + top_k`
- **`update`** — POST `/memory_stores/{store}:update_memories` with `messages + scope`
- **`delete_scope`** — POST `/memory_stores/{store}:delete_scope` with `scope`

Store name defaults to `memory-${SANDBOX_NAME}` (truncated to ≤63 chars for K8s DNS-label compliance), matching the KarsMemory CR `storeName` convention. Operator overrides via `store_name` arg are accepted.

Scope defaults to `agent:${SANDBOX_NAME}`. Operators override with `session:<id>` or `user:<id>` for finer-grained namespacing.

Registration gated on:
- `KARS_PROVIDER` ∉ {`github-models`, `github-copilot`} (slim modes have no Foundry project)
- `FOUNDRY_PROJECT_ENDPOINT` set (otherwise no Foundry to call)

The other 8 Foundry tools (foundry_code_execute, foundry_web_search, foundry_file_search, foundry_image_generation, foundry_conversations, foundry_evaluations, foundry_deployments, foundry_agents, foundry_download_file) reach the LLM via the platform MCP path — see entrypoint change in §5.

13 new tests covering store-name convention, scope default + override, each operation, error handling, registration gating.

### 4. `runtimes/hermes/src/kars_runtime_hermes/plugin/http_fetch.py`

Replaces the A1.1 stub with `http_fetch`:
- POST `/egress/fetch` on the router — subject to blocklist (always), Learn-mode logging (default), allowlist (Strict mode), SSRF protection (localhost/private IPs rejected by `routes/egress.rs:124-144`).
- Response capped at 32 KB to keep prompt-safe.
- 4xx surfaces clear error to the LLM with the router's denial reason.

4 new tests covering URL validation, proxy path, denial surface, response truncation, registration.

### 5. `sandbox-images/hermes/entrypoint.sh` — platform MCP registration

The entrypoint's MCP config generation now ALSO writes a `platform` MCP server entry when `FOUNDRY_PROJECT_ENDPOINT` is set. Hermes' MCP client then discovers all 9 Foundry tools at `http://127.0.0.1:8443/platform/mcp` and exposes them as `mcp:platform:foundry_*` tools alongside the native `foundry_memory` registered in §3.

The translation logic also skips a McpServer CR named `platform` (reserved namespace) to avoid colliding with the built-in.

## Risk Assessment

- **Strictly additive**: All four modules previously shipped as stubs (no-op `register()`). Replacing the stubs activates the kars tool catalogue but doesn't change anything else.
- **Spawn name validation**: Rejects K8s-invalid names BEFORE hitting the router (uppercase, underscores, trailing hyphen, >63 chars). Matches the controller's CRD-side admission validation.
- **Spawn polling has a 45s timeout**: Prevents hung tool calls on slow pod startup. If timeout, returns a `warning` field — operator can poll status separately.
- **Memory store auto-provisioning**: Same retry-once-on-404 pattern OpenClaw uses. If the store doesn't exist, the router auto-creates it on the next call. No new state added.
- **Scope is per-sandbox by default**: Operators must explicitly opt-in to wider scopes (`user:*`) — prevents accidental cross-sandbox memory leaks.
- **HTTP fetch is router-mediated**: SSRF protection + blocklist + allowlist all enforced server-side. The plugin is a thin wrapper — it cannot bypass any of these.
- **No new external dependencies**: Just `httpx` (already a dep).

| Spawn family | router → controller → CRD | router → controller → CRD | router → Docker Engine via `inference-router/src/spawn/docker.rs` (runtime-aware since e105350) |
